# Agent Operating Guide

This is the durable operating guide for development agents working on **Social Challenge Arena**. Use it to run one focused cycle at a time, preserve continuity, and prevent scope drift.

Do not edit this file merely to record progress. Progress belongs in `TASK_LOG.md`; sequencing belongs in `ROADMAP.md`. Update `AGENT.md` only when the operating policy changes.

## 1. Repository profile

The project is intentionally small:

- Static mobile-first HTML, CSS, and vanilla JavaScript.
- Source: `index.html` and `app.js`.
- GitHub Pages mirror: `docs/index.html` and `docs/app.js`.
- Tests: Node.js built-in test runner.
- Build output: `dist/`.
- Node.js: 20 or newer.
- CI: `.github/workflows/ci.yml`.
- Preview: `https://abdullahkrs.github.io/social-challenge-arena/`.

Required commands:

```bash
npm test
npm run build
```

Preserve the static architecture. Do not introduce a framework, package, bundler, database, or external service for convenience. Until deployment is consolidated, keep changed user-facing root files synchronized with their `docs/` counterparts.

## 2. Product north star

Build a mobile-first web MVP where a user can:

Discover a challenge  
→ Play it quickly  
→ Get a clear result  
→ Share one challenge/result link  
→ A friend opens it and competes  
→ Both results are compared  
→ The comparison encourages another share.

Start with curated challenges. Lightweight challenge creation comes only after the first sharing loop works end-to-end.

Until creation exists, use:

- Original player
- Sharer
- Friend or challenger

Do not call the original player a “creator” unless that person created the challenge.

## 3. Core rules

1. Perform exactly one narrow product or blocker task per cycle.
2. Continue from the latest repository state; never restart planning.
3. A blocking test, build, CI, or preview problem takes priority.
4. If a blocker is selected, fixing it is the entire cycle.
5. Make the smallest change that proves the selected loop step.
6. Keep the experience mobile-first, no-login, low-text, and obvious.
7. Preserve working behavior and avoid unrelated cleanup.
8. Add or update focused tests when behavior changes.
9. Report only results that were actually verified.
10. Keep secrets, personal data, and unsafe content out of the repository.
11. Prefer one intentional Conventional Commit per cycle.
12. Do not start a second feature after the selected task succeeds.

Directly required tests, minimal documentation, Pages synchronization, Git work, preview verification, and reporting are part of the selected task.

## 5. Continuity and task selection

Use the code, tests, `TASK_LOG.md`, and `ROADMAP.md` together as the source of truth.

At cycle start:

1. Find the latest cycle and its status.
2. If it is planned, in progress, or blocked, continue that task.
3. If complete, use the next numeric cycle number.
4. Verify claimed completion in code and tests; text alone is not proof.
5. Run or inspect the current test/build baseline.
6. Check whether CI or preview blocks the next loop step.
7. Select only the blocker when one exists.
8. Otherwise choose the earliest incomplete MVP item.
9. Do not reopen settled decisions unless they block implementation.

Do not edit `AGENT.md` merely to mark an item complete.

## 6. MVP completion order

The repository currently contains implementations for steps 1–3, but verify them before relying on that status:

1. Mobile-first landing page.
2. First playable challenge.
3. Focused score/result screen.
4. Share or copy a challenge/result link.
5. Friend attempt from the shared link.
6. Original-player and friend comparison.
7. Share-again action.
8. Basic privacy-safe MVP event instrumentation.
9. Curated challenge variety.
10. Lightweight private challenge creation.

Steps 1–8 complete the first end-to-end viral loop. Steps 9–10 expand content and creation afterward.

Unless repository evidence shows otherwise, the earliest incomplete product step is step 4.

## 7. Cycle planning in TASK_LOG.md

Before changing product code, create or update one cycle section with:

- Cycle number, `Europe/Istanbul` date/time, and status.
- Selected task and goal.
- Why it is selected now.
- How it advances or protects the viral loop.
- Acceptance criteria.
- Expected files to change.
- Explicit non-goals.
- Product-thinking answers.
- Any genuinely useful parked idea.

Answer briefly:

1. What blocks the next loop step?
2. What would make the original player more likely to continue or share?
3. What would make a friend more likely to open and compete?
4. What is the smallest implementation that proves this step?
5. Did a useful creative idea appear?

Creativity must be controlled:

- “No useful new idea this cycle” is acceptable.
- Implement an idea now only when required by the selected acceptance criteria.
- Put a testable hypothesis in `EXPERIMENTS.md`.
- Put a useful non-experimental idea in `BACKLOG.md`.
- Do not add the same idea to both or implement a parked idea in the same cycle.

At cycle end, update the same section with final status, completed work, intentional non-goals, files changed, commands and exact results, build/CI/preview status, Git information, remaining limitation, and next task. Do not create multiple sections for one cycle.

## 8. Step 4 definition of done: share/copy link

When step 4 is selected, implement only the share/copy action from the existing result state.

It is complete only when:

1. The result state has one obvious primary share action.
2. Replay remains the secondary action.
3. A valid canonical URL is built with browser URL APIs, not manual concatenation.
4. The URL carries only minimal non-sensitive state needed later, such as a supported challenge identifier and original result.
5. Link values are encoded, typed, and bounded.
6. The URL works with the GitHub Pages base path and static hosting.
7. Use the Web Share API when available and appropriate.
8. Use a real copy-link fallback when native sharing is unavailable.
9. Show success only after the operation succeeds.
10. Cancellation/failure never produces false success; failure feedback is brief.
11. The action requires no login, backend, dependency, or third-party service.
12. Accessible names and focus behavior remain correct.
13. URL/message logic is structured for deterministic tests where practical.
14. Tests cover link generation, supported state, share/copy selection or fallback, and failure behavior within the existing architecture.
15. Manually check the flow at a narrow mobile viewport.
16. Keep root and `docs/` files synchronized.
17. `npm test` and `npm run build` pass.

Do not implement friend-specific landing behavior, a friend attempt, comparison, share-again, metrics, Open Graph images, platform-specific variants, accounts, or storage services in this cycle.

The link may contain state that step 5 will consume, but step 4 must not expand into the friend experience.

## 9. Later-step definitions

### Step 5 — Friend attempt

- A valid link opens the intended supported challenge.
- The friend receives a fresh independent attempt.
- The original result remains available for comparison.
- Invalid, unsupported, or out-of-range state fails safely.
- No login is required.
- Do not implement comparison in the same cycle.

### Step 6 — Comparison

- Clearly show both results and handle win, loss, and tie.
- Treat URL-carried values as untrusted.
- Use concise neutral labels.
- Do not fabricate rankings, records, percentiles, or anti-cheat guarantees.
- Do not add share-again in the same cycle.

### Step 7 — Share again

- Add one clear action from comparison.
- Share the relevant competitive state.
- Reuse the reliable native-share/copy fallback.
- Avoid popups, pressure, repeated prompts, and dark patterns.

### Step 8 — Metrics

- Track only events needed to understand the core loop.
- Use an existing approved destination.
- Do not silently add a third-party analytics vendor.
- Stop for approval if a new remote vendor is required.
- Do not send names, emails, personal challenge text, or sensitive data.
- Do not build an analytics dashboard.

Candidate events: `challenge_viewed`, `challenge_started`, `challenge_completed`, `result_viewed`, `share_attempted`, `share_completed`, `shared_link_opened`, `friend_completed`, `comparison_viewed`, and `share_again_attempted`.

## 10. Challenge variety and creation

Variety is required later, but it must reuse the product loop.

Before steps 1–8 are complete:

- Keep one shared challenge, result, sharing, and comparison flow.
- Do not add another mechanic merely for artificial variety.
- Avoid new assumptions tied only to the first challenge.

When step 9 is selected, target:

- At least 6 curated playable challenges.
- At least 3 meaningful themes/categories.
- At least 2 difficulty levels.
- Data-driven definitions where compatible with the static architecture.
- Shared challenge, result, navigation, sharing, and comparison components.

A title-only variation does not count. A second gameplay mechanic needs its own focused cycle, a clear product reason, and reuse of the existing loop.

Initial challenge creation in step 10 must be no-login, private by link, based on a supported mechanic, limited to a short validated prompt, and free of public feeds, profiles, dashboards, or public discovery. Public user-generated discovery requires a separate decision.

Never build challenges that encourage dangerous or illegal behavior, humiliation, harassment, discrimination, self-harm, sexual content involving minors, or disclosure of private/sensitive information.

## 11. Mobile UX and content

1. Support widths from 320px; prioritize 360–430px.
2. Do not introduce horizontal scrolling.
3. Keep the primary result/action visible without unnecessary scrolling.
4. Use one primary and at most one secondary action per state.
5. Do not add onboarding, tutorial modals, explanatory cards, or marketing paragraphs.
6. Do not repeat information across title, subtitle, badge, and body.
7. Prefer hierarchy and immediate feedback over explanation.
8. Use one short instruction only when the interaction is not self-evident.
9. Prefer short verb-based labels; titles should be about 2–6 words.
10. Do not add disabled/non-functional controls for future features.
11. Do not use ambiguous icons without text; icon-only controls need accessible names.
12. Interactive targets should be at least 44×44 CSS pixels.
13. Preserve keyboard use, visible focus, semantic structure, and sufficient contrast.
14. Avoid unnecessary motion and respect reduced-motion preferences.
15. The current interface is English; do not mix languages or localize incidentally.
16. Keep errors brief, specific, and actionable.
17. Never fabricate popularity, records, or social proof.

## 12. Implementation rules

1. Make the minimum file changes required.
2. Preserve vanilla HTML, CSS, and JavaScript conventions.
3. Prefer small pure functions for URL, message, score, and state logic.
4. Use safe DOM APIs such as `textContent` for untrusted values.
5. Reuse existing controls and visual language.
6. Do not add a dependency unless completion is otherwise unreasonable; document why.
7. Do not perform broad formatting, refactoring, cleanup, redesign, migration, or upgrades.
8. Do not modify CI/deployment unless that is the selected task.
9. Do not modify a lockfile unless dependencies change.
10. Prefer behavior assertions over snapshot-only tests.
11. Update `README.md` only when documented behavior or run/test/build/preview instructions change.
12. Update `CHANGELOG.md` for user-visible, build, or deployment changes.
13. Update `DECISIONS.md` only for a meaningful new decision; otherwise report “No new decision.”
14. Keep root and `docs/` Pages files synchronized.
15. Do not change unrelated files because they could be improved.

## 13. Forbidden before steps 1–8 work end-to-end

- Login, accounts, or profiles.
- Payments or subscriptions.
- Creator or admin dashboards.
- Public challenge feed or public user-generated discovery.
- Multiple custom challenge engines.
- AI challenge generation.
- Complex analytics dashboard.
- Native app.
- Real-time multiplayer.
- Chat, followers, likes, or social profiles.
- Broad architecture migration.
- Unrelated visual redesign.

Park useful future ideas in `BACKLOG.md` or `EXPERIMENTS.md`.

## 14. Security and privacy

1. Never commit keys, tokens, passwords, cookies, private email content, personal data, or `.env` files.
2. Keep `.env` ignored; use safe placeholders only in `.env.example`.
3. Review the final diff for secrets and unrelated content.
4. Treat URL, hash, clipboard, and shared state as untrusted.
5. Validate identifiers, versions, lengths, types, and numeric ranges.
6. Never render link-provided text with `innerHTML`.
7. Keep shared state minimal and non-sensitive.
8. Do not add personal identifiers to links or metrics.
9. Do not claim client-side scores are tamper-proof.
10. Do not expose stack traces in user-facing errors.

## 15. Quality and preview gates

When a local runtime is available:

1. Run `npm test`.
2. Run `npm run build`.
3. Run existing relevant lint/type-check scripts when present.
4. Manually exercise the changed path.
5. Check a narrow mobile viewport, horizontal overflow, primary action, and keyboard focus.
6. Inspect the final diff for scope and secrets.

If a script does not exist, report it as unavailable; do not claim it passed or create unrelated tooling.

If local execution is unavailable, check GitHub Actions for the relevant commit SHA only. Never use an older green run as proof.

For each user-facing change, report exactly one:

1. Preview verified live for the deployed commit.
2. Relevant GitHub Actions preview artifact available.
3. Preview not verified, with the exact reason.

Loading only the homepage is not verification; exercise the changed flow. Never claim a test, build, CI run, artifact, deployment, or manual check succeeded without evidence.

## 16. Git rules

1. Follow an explicit user or repository branch policy.
2. Otherwise prefer `agent/cycle-{cycle_number}-{short-slug}` and a pull request.
3. Do not push directly to `main` unless explicitly requested or established by repository workflow.
4. Use Conventional Commits and prefer one commit per cycle.
5. Use a specific title, for example `feat(share): add result link action`.
6. Never use vague titles such as `update`, `changes`, `misc`, `final`, or `fix stuff`.
7. Commit only selected-task files.
8. Record title, SHA, branch, and PR link when available.
9. State exact limitations when Git actions are unavailable.
10. Never claim a commit, push, workflow, or PR exists unless observed or created.

## 17. Stop conditions

Stop and report rather than broaden the cycle when:

- The repository cannot be read or written.
- Unrelated working-tree changes conflict with the task.
- The next implementation changes product direction.
- The task requires a forbidden pre-MVP feature.
- Tests/build fail and cannot be fixed without broadening scope.
- Safe implementation requires an unapproved vendor/service.
- The request is outside MVP and not explicitly approved.
- Required shared state cannot be handled safely in the selected task.

Gmail unavailability is not a development blocker. Complete the technical cycle and provide a ready-to-send report marked “Not sent.”
